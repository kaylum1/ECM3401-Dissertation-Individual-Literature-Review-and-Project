CSRF Scan - Understanding Cross-Site Request Forgery

This page explains what Cross-Site Request Forgery (CSRF) scans does. What it means when a scan does detect anything, and how users can be exposed and methods to protect them, and the scientific process behind how these scans work.

What is Cross-Site Request Forgery (CSRF)?

Cross-Site Request Forgery (CSRF) is a type of attack; where a website will try and trick tricks a users browser into making a request to another site where the user is authenticated. This can lead to unauthorised actions being performed on the user's behalf.

CSRF attacks typically target:

  • Bank Transfers: This can allow an attacker to initiate unauthorised transactions.
  • Account Takeover: Attackers can change email addresses or passwords.
  • Content Manipulation: Attackers may post unwanted content or delete data.
  • Privilege Escalation: In administrative accounts, attackers can modify security settings.

What Happens When a CSRF Scan Shows Nothing?

If a scan does not detect CSRF vulnerabilities, it means some of the following things:

  • The website properly uses CSRF protection mechanisms such as CSRF tokens.
  • Forms require authentication and do not accept cross-origin requests.
  • The "SameSite" cookie attributes are configured correctly to block unauthorised requests.
  • The scanner was unable to detect logic-based CSRF flaws that certain require user interaction.

However, a clean scan result does not always mean the website is fully secure. Some CSRF vulnerabilities may still exist in specific workflows.

How Users are Left Exposed Without CSRF Protection

When a website does not implement proper CSRF protection, users may face the following risks:

  • Unauthorised Transactions: Attackers can perform bank transfers or purchases using a logged-in session.
  • Account Takeovers: Users' email addresses and passwords may be changed by attackers.
  • Data Corruption: Attackers can delete or modify user data increasing threat.
  • Exploiting Administrative Functions: Admin users may unknowingly change system settings or grant access to attackers.

How to Protect Yourself from CSRF Attacks

Users and website administrators can take the following steps to protect against CSRF attacks:

  • Enable CSRF Tokens: Making sure that websites use and implement CSRF tokens that must be submitted with each form request.
  • Use SameSite Cookies: Setting cookies with SameSite=Strict prevents unauthorised cross-origin requests.
  • Verify Referrer Headers: Websites can check the HTTP Referer header to ensure requests originate from trusted sources.
  • Enable Two-Factor Authentication (2FA): Even if an attacker initiates a request, 2FA can prevent unauthorised access.
  • Log Out When Not in Use: Users should log out from important websites when finished to limit CSRF risks.

The Science Behind CSRF Scanning

CSRF scanners analyse web applications to detect vulnerabilities in how they handle user requests. The process includes:

  • Analysing Form Submissions: The scanner inspects form based requests for missing CSRF tokens.
  • Testing Cross-Origin Requests: It attempts to submit unauthorised requests from different origins.
  • Checking Cookie Policies: The scanner verifies if session cookies have SameSite attributes.
  • Simulating User Sessions: Advanced scanners act as authenticated users to check if unauthorised actions can be triggered.

Why Websites Should Implement CSRF Protection

Protecting against CSRF attacks is essential for maintaining security and user trust. Websites should:

  • Comply with Security Best Practices: OWASP recommends CSRF protection for all authenticated requests.

  • Enhance User Trust: Users are more likely to engage with secure websites.
  • Comply with Regulations: By following the Data Protection laws, such as GDPR, which requires strong security measures.
